## 💡 "半吊子"的优势与诚实策略

### 为什么"半吊子"反而是优势？

```
【面试官的真实想法】
1. 没有人一开始就是专家
2. 面试官更看重学习能力和解决问题的思路
3. 诚实的候选人比吹牛的候选人更值得信任
4. 能够快速学习新技术的人更有价值

【你的优势】
1. 有实践经验（不是纸上谈兵）
   - 你真的部署过K8s、Docker等
   - 你真的遇到过问题并解决过
   - 这比理论知识更有说服力

2. 有学习能力（能快速成长）
   - 你能通过官方文档学习
   - 你能通过GitHub issues解决问题
   - 你能通过实践不断改进

3. 有诚实态度（值得信任）
   - 你敢于承认不足
   - 你不会吹牛或隐瞒
   - 这让面试官更放心

4. 有成长空间（长期价值）
   - 你还有很多可以学习的地方
   - 你能在工作中快速成长
   - 你能为团队带来新的视角
```

### 诚实的面试策略

```
【核心原则】
1. 不要吹牛，但要自信
2. 承认不足，但展示学习能力
3. 讲述真实经历，而不是假想场景
4. 表达热情，而不是虚假承诺

【具体做法】

当被问到不会的东西时：
❌ 错误："我精通这个技术"（然后被追问时露馅）
✅ 正确："我还没有深入学习这个，但我理解基本概念，
         我相信我能够快速学习。我之前在学习K8s时就是这样做的。"

当被问到做过什么时：
❌ 错误："我部署过完整的OpenStack集群"（实际上只是跟着教程）
✅ 正确："我跟着官方教程部署过K8s集群，虽然还不是专家，
         但我理解了基本的架构和工作流程。"

当被问到遇到过什么问题时：
❌ 错误："我从来没有遇到过问题"（显得不真实）
✅ 正确："我在部署Harbor时遇到了日志驱动问题，
         我通过查看日志和GitHub issues，最终定位到问题原因。"

当被问到性能优化时：
❌ 错误："我进行过深度的性能调优"（可能被追问细节）
✅ 正确："我能够进行基本的性能监控和优化，
         对于复杂的性能问题，我会通过查资料和请教团队来解决。"
```

## 💡 技术问题准备

### Linux 底层机制

**Q: 解释一下 namespace 和 cgroup**

```
【Namespace】
Namespace是Linux内核提供的资源隔离机制，Docker容器就是基于namespace实现的。

主要类型:
- PID namespace: 进程隔离
- NET namespace: 网络隔离
- MNT namespace: 文件系统隔离
- UTS namespace: 主机名隔离
- IPC namespace: 进程间通信隔离
- USER namespace: 用户隔离

实际应用: Docker容器使用namespace实现容器间的隔离

【Cgroup】
Cgroup(Control Groups)是Linux内核提供的资源限制机制。

主要功能:
- CPU限制: 限制进程CPU使用率
- 内存限制: 限制进程内存使用
- IO限制: 限制磁盘IO
- 网络限制: 限制网络带宽

实际应用: 
- Docker使用cgroup限制容器资源
- Kubernetes通过cgroup实现Pod的资源requests和limits
```

**Q: Docker 和虚拟机的区别**

```
【架构层面】
- 虚拟机: 硬件虚拟化，每个VM有完整的OS
- Docker: 操作系统级虚拟化，共享宿主机内核

【资源占用】
- 虚拟机: 资源占用大，启动慢（分钟级）
- Docker: 资源占用小，启动快（秒级）

【隔离性】
- 虚拟机: 隔离性强，安全性高
- Docker: 隔离性相对弱，但足够大多数场景

【使用场景】
- 虚拟机: 需要完整OS、强隔离、多租户
- Docker: 微服务、快速部署、CI/CD
```

### Kubernetes 核心概念

**Q: Kubernetes 的核心组件有哪些？**

```
【Master节点】
1. API Server: 集群的统一入口，所有操作都通过API Server
2. Scheduler: 负责Pod调度，选择合适的Node
3. Controller Manager: 管理各种控制器（Deployment、ReplicaSet等）
4. etcd: 分布式键值存储，保存集群状态

【Node节点】
1. Kubelet: 管理Pod生命周期，与API Server通信
2. Kube-proxy: 实现Service的网络代理和负载均衡
3. Container Runtime: 容器运行时（Docker、containerd等）

【工作流程】
用户提交Pod -> API Server -> Scheduler选择Node -> 
Kubelet创建容器 -> Kube-proxy配置网络
```

**Q: Service 的类型有哪些？**

```
1. ClusterIP: 集群内部访问（默认）
2. NodePort: 通过节点端口暴露服务
3. LoadBalancer: 云厂商负载均衡器
4. ExternalName: 映射到外部DNS

实际使用:
- 内部服务用ClusterIP
- 需要外部访问用NodePort或LoadBalancer
- 我在项目中使用NodePort暴露服务供外部访问
```

### Docker 相关

**Q: Dockerfile 的最佳实践**

```
1. 使用小的基础镜像（alpine）
2. 合并RUN命令减少层数
3. 使用.dockerignore排除不需要的文件
4. 多阶段构建减小镜像大小
5. 不要在镜像中存储敏感信息
6. 使用非root用户运行

示例:
FROM golang:1.20-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o main .

FROM alpine:latest
RUN adduser -D appuser
USER appuser
COPY --from=builder /app/main /app/main
CMD ["/app/main"]
```

**Q: Docker 网络模式**

```
1. bridge: 默认模式，容器间通过虚拟网桥通信
2. host: 容器使用宿主机网络
3. none: 容器没有网络
4. container: 容器共享另一个容器的网络
5. overlay: 跨主机容器通信（Swarm/K8s）

实际应用:
- 大多数场景用bridge
- 需要高性能用host
- K8s使用overlay实现跨节点Pod通信
```
